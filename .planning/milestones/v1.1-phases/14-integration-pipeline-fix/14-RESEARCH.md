## Summary

Phase 14 is a targeted gap-closure phase that fixes two integration breaks identified in the v1.1 milestone audit, removes the per-plugin timezone field in favour of user-level timezone, adds a retry button for failed tiles, and cleans up dead code in touched files.

**The core problem (CREW-05):** The Python sidecar in `sidecar/executor.py` (line 55) returns `result.raw` — a raw Markdown string from CrewAI's workflow output. The Go handler in `internal/streams/handler.go` (line 35) passes this directly to `datatypes.JSON()`. PostgreSQL's JSONB column rejects non-JSON content, leaving PluginRun in the `processing` status permanently. The fix is in the sidecar: wrap the output before publishing, producing `{"summary": "...", "sections": [...]}`.

**The second problem (SCHED-04):** The scheduler in `internal/worker/scheduler.go` reads `cfg.Timezone` (per-plugin) and falls back to UTC when empty. `User.Timezone` is populated by browser auto-detection via `POST /api/user/timezone` → `UpdateTimezoneHandler` but is never consulted. The decision is to eliminate the per-plugin `Timezone` field entirely and read `User.Timezone` from the already-preloaded `User` association in `handlePerMinuteScheduler`.

**The approach:** The sidecar change is ~30 lines of Python. The Go-side output format change is backward-incompatible by design (old runs show as unavailable). The timezone change requires a DB migration to drop the column, a model update, a scheduler update, and settings UI cleanup. Dead code items (ValidateUserSettings, TileSkeleton, stale comment) are surgical file edits.

**Primary recommendation:** Implement in two plans: Plan 1 — output format contract fix (sidecar + Go handler + extractSummary/extractContent + retry button + TileSkeleton/ValidateUserSettings dead code). Plan 2 — timezone migration (DB migration + model + scheduler + settings UI removal + user-level settings page addition + stale comment).

## Architecture Patterns

### Pattern 1: Sidecar JSON Wrapping (Python — sidecar/executor.py)

**What:** Instead of returning `result.raw` (Markdown string), the executor wraps CrewAI output in a structured JSON object before returning `PluginResult`.

**Current code (broken):**
```python
# sidecar/executor.py:53-59
result = await crew.kickoff_async(inputs=request.settings)
output = result.raw if hasattr(result, 'raw') else str(result)
return PluginResult(
    plugin_run_id=request.plugin_run_id,
    status="completed",
    output=output  # ← raw Markdown string, fails JSONB
)
```

**Fixed pattern:**
```python
import json

result = await crew.kickoff_async(inputs=request.settings)
raw_output = result.raw if hasattr(result, 'raw') else str(result)

# Generate summary: first paragraph or first N chars of raw output
summary = _extract_summary(raw_output)

# Build sections array from raw output
sections = _build_sections(raw_output)

# Wrap as valid JSON — this is what Go side stores in JSONB
structured = json.dumps({
    "summary": summary,
    "sections": sections
})

return PluginResult(
    plugin_run_id=request.plugin_run_id,
    status="completed",
    output=structured  # ← valid JSON string
)
```

**Section structure (Claude's discretion recommendation):** Given CrewAI produces Markdown, split on `##` headings. Each section has `{"title": str, "content": str}`. If no headings found, produce a single section `{"title": "Briefing", "content": raw_output}`. This is simple, lossless, and works with any CrewAI output format.

```python
def _build_sections(raw: str) -> list[dict]:
    """Split Markdown output on ## headings into sections."""
    import re
    parts = re.split(r'^##\s+', raw, flags=re.MULTILINE)
    if len(parts) <= 1:
        return [{"title": "Briefing", "content": raw.strip()}]
    sections = []
    for part in parts[1:]:  # skip content before first heading
        lines = part.split('\n', 1)
        title = lines[0].strip()
        content = lines[1].strip() if len(lines) > 1 else ""
        sections.append({"title": title, "content": content})
    return sections

def _extract_summary(raw: str) -> str:
    """Extract first paragraph as summary (≤300 chars)."""
    paragraphs = [p.strip() for p in raw.split('\n\n') if p.strip()]
    # Skip heading lines (start with #)
    for p in paragraphs:
        if not p.startswith('#'):
            return p[:300]
    return raw[:300]
```

**Malformed output fallback (Claude's discretion recommendation):** If the CrewAI output is empty or cannot be processed, wrap it rather than fail: `{"summary": "Briefing generated", "sections": [{"title": "Content", "content": raw}]}`. Reserve `status="failed"` for genuine CrewAI failures (exceptions, timeouts) which already publish `{"error": "..."}`.

### Pattern 2: Go Side — PluginRunOutput JSON Format Change

**What:** `PluginRunOutput` in `internal/dashboard/viewmodel.go` already has `Summary string` and `Content string`. The user decision changes the format to `{summary, sections[]}`. The `Content` field in `PluginRunOutput` must be updated to handle `sections[]`.

**Key insight:** `extractContent` currently returns `out.Content` (a string). With `sections[]`, `extractContent` should concatenate sections into a single HTML string for tile rendering. The `BriefingContent` in `TileViewModel` is rendered via `@templ.Raw()` so HTML is acceptable.

**Updated PluginRunOutput struct:**
```go
// PluginRunOutput is the JSONB structure stored in plugin_runs.output.
type PluginRunOutput struct {
    Summary  string          `json:"summary"`
    Sections []OutputSection `json:"sections"`
    // Legacy Content field: kept for parsing only, deprecated in new output
    Content string `json:"content"`
}

// OutputSection is a titled section within a plugin run's output.
type OutputSection struct {
    Title   string `json:"title"`
    Content string `json:"content"`
}
```

**Updated extractContent:**
```go
func extractContent(output []byte) string {
    if len(output) == 0 {
        return ""
    }
    var out PluginRunOutput
    if err := json.Unmarshal(output, &out); err != nil {
        return ""
    }
    // New format: build HTML from sections
    if len(out.Sections) > 0 {
        var sb strings.Builder
        for _, s := range out.Sections {
            if s.Title != "" && s.Title != "Briefing" {
                sb.WriteString("<h3>" + s.Title + "</h3>")
            }
            sb.WriteString("<p>" + s.Content + "</p>")
        }
        return sb.String()
    }
    // Legacy: single content field (old runs)
    return out.Content
}
```

**extractSummary:** Already correct — returns `out.Summary` if non-empty, falls back to first 200 chars of Content. Works with new format unchanged.

## Common Pitfalls

### Pitfall 1: PluginResult.output type in Python model (sidecar/models.py)

**What goes wrong:** `PluginResult.output` is typed `str | None`. The wrapping produces a JSON string (a `str` Python type). This is correct — the string contains valid JSON. The Go side `json.Unmarshal([]byte(payloadStr), &result)` then parses `result.Output` as a Go string. Then `datatypes.JSON(result.Output)` is stored as JSONB. PostgreSQL is happy because `result.Output` is a valid JSON string.

**Why it happens:** Confusion between "JSON as a string" vs "JSON object". The sidecar stores the structured output as a JSON-encoded string. The stream message wraps it again: the outer is `{"plugin_run_id": ..., "status": "completed", "output": "{\"summary\":...}"}`.

**How to avoid:** Do NOT change `PluginResult.output` to `dict` — that would break the Pydantic `model_dump_json()` serialization into the stream. Keep `output: str | None` and assign `json.dumps({...})` to it.

### Pitfall 2: datatypes.JSON() expects valid JSON bytes

**What goes wrong:** `datatypes.JSON(result.Output)` in `internal/streams/handler.go` — `result.Output` is a Go `string`. `datatypes.JSON` is `[]byte`. If the string is not valid JSON, PostgreSQL will reject it with `invalid input syntax for type json`.

**How to avoid:** After the sidecar fix, `result.Output` will always be a valid JSON string (from `json.dumps()`). No change needed to the Go handler. Optionally add a validation check: `json.Valid([]byte(result.Output))` before storing — log warning and mark run failed if invalid.

### Pitfall 3: XReadGroup/`>` vs pending delivery of malformed messages

**What goes wrong:** If there are existing messages in the `plugin:results` stream from before this fix, the Go consumer will try to process them. `result.Output` will be raw Markdown, `extractSummary` and `extractContent` will return `""` (JSON parse fails), and the tile will show as completed with no content.

**How to avoid:** The consumer's `extractSummary`/`extractContent` returns `""` on parse failure — the tile shows the "waiting" state even if status=completed. The user can press Retry. Old runs are acceptable to leave as-is (no backwards compatibility per decision). No stream purge needed.

### Pitfall 4: Removing Timezone from UserPluginConfig — references in SaveSettingsHandler

**What goes wrong:** `SaveSettingsHandler` reads `timezone := rawForm.Get("timezone")` and sets `config.Timezone`. If the model field is removed but the handler code remains, GORM will error when trying to save a field that doesn't exist on the struct.

**How to avoid:** Remove ALL references to `config.Timezone` in:
- `internal/settings/handlers.go` (SaveSettingsHandler: two locations where `config.Timezone` is set)
- `internal/settings/viewmodel.go` (SQL query with `COALESCE(upc.timezone, 'UTC') AS timezone`, `pluginRow.Timezone`, `vm.Timezone`)
- `internal/settingsvm/settingsvm.go` (`PluginSettingsViewModel.Timezone string`)
- `internal/templates/settings.templ` (`renderTimezoneSelect` call in PluginSettingsForm)
- `internal/dashboard/viewmodel.go` (`configRow.Timezone`, SQL query `upc.timezone`)

Also: `TogglePluginHandler` creates new `UserPluginConfig` with `Timezone: "UTC"` — remove this field initialization.

### Pitfall 5: computeNextRun uses timezone from configRow

**What goes wrong:** `computeNextRun(cfg.CronExpression, cfg.Timezone)` is called in both `getDashboardTiles` and `GetSingleTile`. If `configRow.Timezone` is removed from the SQL query, `cfg.Timezone` will be empty, causing all cron schedules to evaluate in UTC regardless of user preference.

**How to avoid:** Replace `upc.timezone` with `u.timezone` (from the users table JOIN) in the SQL queries in `getDashboardTiles` and `GetSingleTile`. The field name on `configRow` can stay as `Timezone string` — just source it from the users table.

Similarly in `internal/settings/viewmodel.go`, `getPluginStatus` calls `computeNextRun(cronExpr, timezone)` with the timezone from `pluginRow.Timezone`. The settings viewmodel query must also join users to get `u.timezone`.

### Pitfall 6: TileSkeleton removal — compiled _templ.go file

**What goes wrong:** Removing `TileSkeleton` from `dashboard.templ` without running `make templ-generate` will leave a stale function in `dashboard_templ.go`. Go will still compile because `_templ.go` files are auto-generated but not auto-regenerated. If `TileSkeleton` is referenced nowhere, the stale generated code is harmless but confusing.

**How to avoid:** After removing from `.templ`, always run `make templ-generate`. This is already the project standard workflow.

### Pitfall 7: User-level settings page route conflicts

**What goes wrong:** The existing sidebar sub-link for Settings points to `/settings/plugins`. Adding `/settings/account` requires both the route AND the sidebar sub-link. If the route exists but sidebar doesn't show it, users can't find it.

**How to avoid:** Update both `routes` (in main.go) AND `AppSidebar` in `settings.templ` to include the new "Account" sub-link under Settings.

### Sidecar — Full Updated executor.py execute() method

```python
async def execute(self, request: PluginRequest) -> PluginResult:
    crew = self._load_crew(request.plugin_name, request.settings)
    if crew is None:
        return PluginResult(
            plugin_run_id=request.plugin_run_id,
            status="failed",
            error=f"Plugin crew not found: {request.plugin_name}"
        )

    try:
        async with asyncio.timeout(self.timeout_seconds):
            result = await crew.kickoff_async(inputs=request.settings)
            raw_output = result.raw if hasattr(result, 'raw') else str(result)

            # Wrap as structured JSON before returning
            structured = self._wrap_output(raw_output)

            return PluginResult(
                plugin_run_id=request.plugin_run_id,
                status="completed",
                output=structured  # valid JSON string
            )
    except asyncio.TimeoutError:
        return PluginResult(
            plugin_run_id=request.plugin_run_id,
            status="failed",
            error=f"Workflow exceeded {self.timeout_seconds}s timeout"
        )
    except Exception as e:
        return PluginResult(
            plugin_run_id=request.plugin_run_id,
            status="failed",
            error=str(e)
        )

def _wrap_output(self, raw: str) -> str:
    """Wrap raw CrewAI output as structured JSON string."""
    import json, re
    summary = self._extract_summary(raw)
    sections = self._build_sections(raw)
    return json.dumps({"summary": summary, "sections": sections})

def _extract_summary(self, raw: str) -> str:
    paragraphs = [p.strip() for p in raw.split('\n\n') if p.strip()]
    for p in paragraphs:
        if not p.startswith('#'):
            return p[:300]
    return raw[:300]

def _build_sections(self, raw: str) -> list:
    import re
    parts = re.split(r'^##\s+', raw, flags=re.MULTILINE)
    if len(parts) <= 1:
        return [{"title": "Briefing", "content": raw.strip()}]
    sections = []
    for part in parts[1:]:
        lines = part.split('\n', 1)
        title = lines[0].strip()
        content = lines[1].strip() if len(lines) > 1 else ""
        sections.append({"title": title, "content": content})
    return sections
```

### Go — Updated PluginRunOutput (internal/dashboard/viewmodel.go)

```go
type PluginRunOutput struct {
    Summary  string          `json:"summary"`
    Sections []OutputSection `json:"sections"`
    Content  string          `json:"content"` // legacy — old completed runs only
}

type OutputSection struct {
    Title   string `json:"title"`
    Content string `json:"content"`
}

func extractContent(output []byte) string {

        return ""
    }

}
```

## Open Questions

1. **Markdown rendering in sections content**
   - What we know: `extractContent` returns a string passed to `@templ.Raw()` in the tile. CrewAI content is Markdown.
   - What's unclear: Whether raw Markdown text in `<p>` tags looks acceptable or broken (depends on whether content has list items, bold, etc.).
   - Recommendation: Wrap each section's content in a `<pre class="tile-content-section">` or plain `<p>` tag for MVP. Full Markdown-to-HTML rendering (e.g., `goldmark`) is a future improvement. The tile is already constrained in size; users will see the content in the expanded view.

2. **ValidateUserSettings callers**
   - What we know: `grep -rn "ValidateUserSettings"` found zero call sites (only the definition). Safe to delete the file.
   - What's unclear: Nothing — confirmed dead code.
   - Recommendation: Delete `internal/plugins/validator.go` entirely.

3. **computeNextRun in settings/viewmodel.go — timezone source**
   - What we know: `getPluginStatus` calls `computeNextRun(cronExpr, timezone)` with `pluginRow.Timezone` (currently from `COALESCE(upc.timezone, 'UTC')`). After removing the column, must source from `u.timezone`.
   - What's unclear: Whether the `getPluginStatus` query already joins users table.
   - Current query (line 94-111 of `settings/viewmodel.go`): Does NOT join users table. Must add `JOIN users u ON u.id = upc.user_id` and select `u.timezone`.
   - Recommendation: Add `UserTimezone string` to `pluginRow` struct and update the SQL query.

4. **Account settings page — scope for this phase**
   - What we know: User decision is to add timezone picker to user-level settings. This requires a new sub-page.
   - What's unclear: Whether to create a full account settings hub or just a minimal timezone-only page.
   - Recommendation: Create a minimal `/settings/account` page with just the timezone picker for this phase. Additional account settings (name, email) are deferred.
