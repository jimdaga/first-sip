# Phase 16 Plan 02: API Key Management UI Summary

**Glass-morphism API Keys settings page with HTMX fragment swaps for add/delete LLM and Tavily keys, client-side model filtering, and LLM preference persistence — wired to sidebar and settings hub**

## Accomplishments

- `internal/apikeysvm/types.go` — leaf package with `APIKeysPageViewModel`, `APIKeyViewModel`, `LLMProviderVM` to prevent import cycle between `apikeys` and `templates`
- `internal/apikeys/viewmodel.go` — `BuildViewModel` converts service data into display models with masked key values and human-readable provider names
- `internal/apikeys/handlers.go` — four Gin handlers: `PageHandler`, `SaveKeyHandler`, `DeleteKeyHandler`, `SaveLLMPreferenceHandler`; all return HTMX fragment swaps on success
- `internal/templates/settings.templ` — six new/updated components: sidebar API Keys sub-link, settings hub tile, `APIKeysSettingsPage`, `APIKeysSection`, `LLMPreferenceSection`, error alert fragments
- Routes registered in `cmd/server/main.go` protected group

## Files Created/Modified

## Decisions Made

- **apikeysvm leaf package**: `templates` imports `apikeysvm`, `apikeys` imports `apikeysvm` and `templates`. No cycle. Follows the same pattern as `settingsvm`.
- **Client-side JS model filtering**: When the user changes the provider select in LLM Preferences, JavaScript reads an embedded JSON map to filter the model select without an HTMX round-trip. The `providerModelsJSON()` helper function serializes `vm.Providers` to JSON at render time.
- **Error alert fragments replace the full section div**: `APIKeysErrorAlert` and `LLMPreferenceErrorAlert` return `<div id="api-keys-section">` and `<div id="llm-preference-section">` respectively, so HTMX `outerHTML` swap works consistently with the success path.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Structure] Introduced apikeysvm leaf package to break import cycle**
- **Found during:** Task 1 — discovered that `apikeys/handlers.go` importing `templates` and `templates/settings.templ` importing `apikeys` would create a circular import
- **Issue:** Go does not allow circular imports. Plan assumed handlers and templates could mutually import, which is not possible.
- **Fix:** Created `internal/apikeysvm/types.go` with view model types (no internal imports). Templates import `apikeysvm`. `apikeys` imports `apikeysvm` + `templates`. No cycle.
- **Files modified:** `internal/apikeysvm/types.go` (new), `internal/apikeys/viewmodel.go` (updated to return apikeysvm types)
- **Pattern:** Same as existing `settingsvm` package used by `settings` handlers + `templates`.
